# TheBible

A Claude Code plugin that creates and maintains a **Bible** — a comprehensive single source of truth document — for any codebase.

## Why

Documentation drifts from code. READMEs go stale. CLAUDE.md captures operational shortcuts but not the full picture. TheBible solves this with:

- A **Bible** — one markdown file that comprehensively documents your entire system (architecture, database, APIs, frontend, auth, testing, deployment, patterns, gotchas)
- An **automated triage engine** that detects what code changed since the last Bible update, maps changes to affected sections, and applies targeted edits
- **CLAUDE.md sync** — operational content from the Bible automatically flows into CLAUDE.md via the `claude-md-management` plugin

The result: a 3-layer documentation system where the Bible is the deep truth, CLAUDE.md is the operational quick-reference, and Memory handles conversational context. All kept in sync automatically.

## Install

```
/plugin marketplace add DantesPeak85/the-bible
```

## Skills

### `/init-bible` — Bootstrap

Creates a new Bible for your project:

1. Scans your codebase to detect tech stack (Node, Python, Rust, Swift, Go, etc.)
2. Maps your directory structure to 14 default documentation sections
3. Generates a `.bible.yaml` config file with path-to-section mappings
4. Launches exploration agents to read your source code
5. Generates a substantive Bible with real content (not just headers)
6. Optionally syncs operational content to CLAUDE.md

### `/update-bible` — Keep it current

Three-phase workflow:

**Phase 1 — Triage:** Parses the Bible's version date, runs `git log` since then, categorizes every changed file against your `.bible.yaml` section mappings, scores impact (HIGH/MEDIUM/LOW), and presents a summary. You approve before any edits happen.

**Phase 2 — Update:** Explores changed areas in detail, reads only the affected Bible sections, applies targeted edits (never rewrites the whole file), bumps the version, and commits.

**Phase 3 — CLAUDE.md Sync:** Automatically invokes `claude-md-management:claude-md-improver` to sync operational changes into CLAUDE.md.

### `/bible-status` — Quick check

Read-only staleness report:

```

============

Version: 2.3 (March 15, 2026)
Age: 29 days

Changes since last update: 47 commits

  API & Services: 15 commits
  Frontend / Client: 12 commits

Staleness: HIGH
Recommendation: Run /update-bible
```

## Configuration

TheBible uses a `.bible.yaml` file in your project root. Generated by `/init-bible`, editable by hand.

```yaml
bible_path: "docs/BIBLE.md"
version: "1.0"
last_updated: "2026-04-13"

project:
  name: "MyProject"
  description: "One-line description"

claude_md:
  enabled: true
  path: "CLAUDE.md"
  operational_sections:
    - "Project Overview"
    - "Build & Deploy"
    - "Key Patterns & Conventions"
    - "Gotchas & Pitfalls"

sections:
  - number: 1
    name: "Project Overview"
    paths: []
  - number: 6
    name: "API & Services"
    paths: ["api/", "routes/", "functions/", "services/"]
  # ... see references/bible-yaml-schema.md for full schema
```

## Default Bible Sections

| # | Section | Purpose |
|---|---------|---------|
| 1 | Project Overview | What, who, why |
| 2 | Architecture Overview | System diagram, components |
| 3 | Tech Stack | Languages, frameworks, versions |
| 4 | Data Flow | How data moves through the system |
| 5 | Database & Storage | Schema, migrations, storage |
| 6 | API & Services | Endpoints, functions, microservices |
| 7 | Frontend / Client | UI, views, state management |
| 8 | Authentication & Security | Auth flows, permissions |
| 9 | Testing | Strategy, frameworks, commands |
| 10 | Build & Deploy | Build commands, CI/CD, deployment |
| 11 | Environment & Configuration | Env vars, config files |
| 12 | Key Patterns & Conventions | Naming, coding patterns |
| 13 | Gotchas & Pitfalls | Non-obvious behaviors, known issues |
| 14 | Glossary | Project-specific terms |

Sections with no relevant code detected are skipped. Add custom sections for domain-specific needs (ML pipelines, compliance, integrations, etc.).

## How It Works

```
Code changes → git log → triage engine → section mapping → targeted edits → version bump → CLAUDE.md sync
```

The triage engine is the core innovation: instead of periodically rewriting documentation, it detects *exactly which sections* are stale via git-diff path matching against your `.bible.yaml` config. This means:

- Only affected sections get updated (fast, focused)
- You see exactly what changed before approving (no surprises)
- Version history tracks what each update covered
- CLAUDE.md stays in sync automatically

## Requirements

- Claude Code CLI
- Git repository
- Optional: `claude-md-management` plugin (for CLAUDE.md sync)
